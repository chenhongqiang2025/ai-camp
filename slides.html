AI 发展简史

从图灵的疑问，到达特茅斯的命名，到两次寒冬，到机器学习的复兴 —— AI 走了 70 年

IN DESIGN 15 min
1950

图灵的疑问

艾伦 · 图灵发表论文 Computing Machinery and Intelligence，提出"机器能不能思考"这个问题，并给出后来被称为"图灵测试"的判断方法 —— 如果一台机器能让人在对话中分不清它是人还是机器，那就有理由说它"思考"

1950 年，图灵就问出了那个直到今天还在问的问题——机器能不能思考。 他给出一个判断方法，后来叫"图灵测试"——如果一台机器跟你聊天，你分不清它是人还是机器，那就有理由说它"思考"了。 今天我们用 ChatGPT 跟它聊天，本质上就是图灵 75 年前预言的那个场景。
1956

达特茅斯会议 · AI 正式命名

John McCarthy、Marvin Minsky、Claude Shannon 等学者在达特茅斯学院召开为期 8 周的研讨会 —— "Artificial Intelligence" 这个词第一次出现在公开学术议程里

1956 年夏天，几位学者在达特茅斯学院开了一个为期 8 周的研讨会——"Artificial Intelligence"这个词，第一次出现在公开学术议程里。 参会的几位都是当时计算机学界的核心人物——McCarthy 后来发明了 LISP；Minsky 是 MIT AI 实验室的创始人；Shannon 是信息论的奠基人。 所以 AI 不是这两年才有的词，已经 70 年了。

  • 核心矛盾：硬规则无法应对真实世界的不确定性

  • 资本和媒体全面撤退
两次寒冬都源于资金和媒体撤离 —— 当时的方法（符号推理、专家系统）解决不了真实世界的不确定性
AI 走到今天不是一条直线，至少冬过两次。 第一次是 70 年代——那一拨学者承诺 10 年内造出通用智能，做不到，资金撤了。 第二次是 80 年代末——专家系统泡沫破裂，LISP 机商业失败，日本"第五代计算机"国家级项目最终也没出成果。 两次寒冬有一个共同原因——当时的方法解决不了真实世界的不确定性。今天的 GenAI 是第三次起。

CHAPTER · 03

机器学习与神经网络

从一个"神经元"开始，到能识别万物的深度网络 —— 数学并不可怕

IN DESIGN 20 min
CH.03 · 01 / 04 · 起点

感知机 · 第一个人工神经元

1957 年 Frank Rosenblatt 提出 — 把生物神经元抽象成一行数学

  • 输入
    x₁ · x₂ · x₃ ... — 一组数字，可以是像素、词的索引、传感器读数本质：把世界翻译成"数"
  • 权重
    w₁ · w₂ · w₃ ... — 每个输入的"重要程度"，由训练得来这是网络"记忆"的全部 — 所谓"AI 模型"就是一堆 w
  • 求和
    z = Σ wᵢxᵢ + b — 加权求和，再加上一个偏置项中学数学就能看懂的式子
  • 激活
    y = σ(z) — 非线性函数（Sigmoid、ReLU），决定神经元是否"激活"没有这一步，整个网络等价于一个线性回归
今天的大模型 —— 数百亿到上万亿个这样的"神经元"叠起来，单个神经元的数学不超出中学范围
神经网络听起来玄，单个神经元的结构其实非常简单。 输入几个数，每个数乘一个权重，加起来，过一个非线性函数，输出一个数。仅此而已，中学数学就够看懂。 那为什么这么简单的东西能做出 ChatGPT？ 因为可以堆。今天最大的模型，有数千亿到上万亿个这样的神经元叠在一起。复杂性来自规模和组合，不来自单点。

神经网络是怎么"学"的？ 先猜一次，看跟正确答案差多远，然后把这个"错"沿着网络反向传回去，告诉每个参数应该往哪边调一点。重复几百万次，它就慢慢调到位了。 数学上叫梯度下降加链式法则，今天不展开。 你只要记住一句话——神经网络不是被编程的，是被"调"出来的。
CH.03 · 03 / 04 · 看图

CNN · 教计算机"看"

1998 年 Yann LeCun 的 LeNet — 用"卷积"模拟人眼的局部感知

传统全连接

每个像素接每个神经元

  • 参数爆炸：100×100 图像就是 10000 个输入
  • 不利用图像的"空间局部性"
  • 训练慢、效果差
卷积神经网络

用小窗口扫描整张图

  • 卷积层：3×3 窗口提取局部特征
  • 池化层：压缩信息，保留要点
  • 参数共享：同一卷积核扫整图
  • 结果：训练快 10×，准确率高 5-10%
AlexNet 2012 在 ImageNet 把 top-5 错误率从 26.2% 拉到 15.3% —— 之后人脸识别、自动驾驶、医疗影像都建立在 CNN 之上
CNN 的核心想法很朴素——图像有"局部性"，旁边的像素往往是相关的。 所以不需要每个像素都跟每个神经元相连，用一个小窗口扫整张图就够了。这样参数少很多，训练快很多。 2012 年 AlexNet 把 CNN 加上 GPU，在 ImageNet 大赛把 top-5 错误率从 26.2% 砍到 15.3%——计算机视觉的拐点。 一个有意思的事实——CNN 1998 年 LeCun 就做出来了，等了 14 年硬件才跟上。这种"算法等硬件"的故事，在 AI 里反复出现。
CH.03 · 04 / 04 · 序列

RNN · 教计算机"记上下文"

CNN 让计算机看图，RNN 让计算机处理一切有顺序的数据 —— 语言、语音、时序

  • 结构
    循环 · 每一步的输出回喂下一步网络第一次有了"上下文"的概念
  • 应用
    机器翻译 · 语音识别 · 文本生成Google Translate 早期就是基于 LSTM（RNN 的改进版）
  • 短板
    长序列梯度消失 · 串行不可并行长句子容易丢信息；训练慢；GPU 利用率上不去
RNN 的两个根本问题（长依赖丢失、不可并行），正是 2017 年 Transformer 要解决的
CNN 让计算机会"看图"，RNN 让计算机会处理"有顺序"的数据——语言、语音、时序。 RNN 的结构很简单——每一步的输出会回喂到下一步，所以网络有了"上下文"的概念。早期的机器翻译、语音识别基本都靠它。 但 RNN 有两个迈不过去的坎——长句子记不住远处的词，而且没法并行训练，GPU 利用率上不去。 这两个坎，下一章 Transformer 用 Attention 一次解决了。
CHAPTER · 04

NLP 演进史

让机器"理解"语言：从词袋到 Transformer 的范式跃迁

IN DESIGN 20 min
CH.04 · 01 / 04 · 起点

词袋模型 · 最朴素的开始

2000 年代 NLP 主流方法 — 把一句话当成一袋词，统计频率

优点

简单粗暴有效

  • 实现 10 行代码
  • 可解释（每个词的权重清晰）
  • 当年的垃圾邮件分类、新闻分类基础
致命缺陷

丢失了"顺序"和"语义"

  • "我爱你" 和 "你爱我" 在词袋里完全一样
  • "猫" 和 "犬" 在向量上没有任何相似性
  • 同义词、反义词都识别不出来
  • 遇到长文本，向量稀疏到几乎全是 0
词袋丢失了语序、也无法表达词与词的语义关系 —— 后续 20 年 NLP 都在解这两个问题
NLP 最朴素的方法叫词袋——把一句话拆成一堆词，统计每个词出现几次。 简单粗暴有效，10 行代码就能写出来，当年的垃圾邮件分类、新闻分类基本都用它。 但它有两个致命问题——"我爱你"和"你爱我"，在词袋里完全一样，丢了顺序。"猫"和"犬"，在词袋里也完全没关系，丢了语义。 这两个问题，是后面 NLP 20 年要解的两道题。
CH.04 · 02 / 04 · 突破

Word2Vec · 把"语义"变成几何

2013 年 Mikolov 在 Google —— 让每个词成为一个高维向量

king − man + woman ≈ queen

每个词放进一个几百维的向量空间，相似的词彼此靠近 —— 性别、复数、时态这些语义关系，变成了空间中的"方向"

今天 RAG 用的"语义检索"，本质就是在 Word2Vec 这一脉的 embedding 空间里做近邻搜索
2013 年，Google 的 Mikolov 提出 Word2Vec，让每个词变成一个几百维的向量。 奇迹出现——你在这个空间里做减法，king 减 man 加 woman，结果非常接近 queen。 (可以让大家现场猜一个——tokyo 减 japan 加 france 等于什么？答案是 paris。) Word2Vec 第一次让"语义"变成可以加减乘除的几何关系。 今天 RAG 里的"向量检索"，根上就是从这里来的。
CH.04 · 03 / 04 · Attention

注意力机制 · 让模型自己挑重点

2014 年 Bahdanau 在机器翻译中引入 —— 输出每个词时，回头看输入哪些词最相关

  • 问题
    RNN 翻长句子会丢信息整句话被压成一个固定向量，远处的词容易被淡忘
  • 想法
    翻每个输出词时，给每个输入词一个"注意力权重"翻 tea 时，"茶"的权重高；翻 rainy 时，"雨天"的权重高
  • 效果
    2015–2017 机器翻译质量明显提升Attention 在 NLP 圈成为标配模块
Attention 解决了"远处信息怎么留住"这个 RNN 的核心难题 —— 三年后被 Transformer 推到中心位置
RNN 翻长句子的瓶颈是——整句话被压成一个固定大小的向量，远处的词容易丢。 2014 年 Bahdanau 提出一个想法——翻每一个输出词的时候，回头看一遍原句，给原句每个词一个"我现在该多关注它"的分数。 翻 tea 的时候，"茶"这个词的分数高；翻 rainy 的时候，"雨天"的分数高。 这个简单的想法，三年后被 Transformer 推到极致——整个 Transformer 就是反复在做 Attention。

CH.06 · 02 / 03 · 根因

为什么大模型会幻觉

这是架构带来的，不是某家做得不好

机制 · 逐 token 生成

每一个字都是概率挑出来的

  • 大模型工作方式：根据上下文预测下一个 token
  • 选的不是"事实"，是"概率最高的下文"
  • 训练数据里没有的事，它会按语言习惯补一个"听起来对"的
  • 模型本身没有"真实世界"的概念
知识 · 压缩进权重

训练数据是过去的快照

  • 每个模型都有训练数据截止日期
  • 训练把数据压缩进神经网络的几百亿权重
  • 压缩有损 —— 罕见的事实细节会丢失或混淆
  • "今天的事"模型基本没有
这是架构问题，不是再训练几轮就能解决的 —— 这就引出了下一章 RAG
为什么大模型会幻觉？两个原因。 第一，它的工作机制是逐 token 生成——每个字都是按"上下文里下一个字概率最高是什么"挑出来的。它没有"事实/不事实"的开关，只有"概率高/低"。 第二，它的知识是训练时压进神经网络权重里的——有截止日期、有损失、罕见的事实容易模糊。 这两件事决定了一件事——让大模型凭记忆回答事实问题，本身就是用错了它。 所以 RAG 不是"让它变聪明"，是"换一种用法"——回答前先去查资料。下一章就讲这个。

MCP 由 Anthropic 在 2024.11 提出 —— 之后 OpenAI、Google、IBM 等相继支持，事实上成为大模型接外部工具的开放标准
2024 年之前，每家 AI 厂自己有一套让模型接工具的方式——OpenAI 用 Function Calling，Anthropic 用 Tool Use，各种 Agent 框架自己又造一套。 结果每个工具都要为每家 AI 写适配——N 乘 M 的灾难。 2024 年 11 月，Anthropic 提出 MCP——一个开放协议，不绑定任何一家。 之后 OpenAI、Google、IBM 都跟进了。一个协议站住了，整个生态就开始受益——从 N 乘 M 变成 N 加 M。 注意一个用词区别——MCP 不是又一个 API，它是协议。API 是接口，协议是大家都遵守的"说话方式"。

MCP 分三个角色——Host、Client、Server。 Host 是你直接用的 AI 应用——Claude Desktop、Cursor、VSCode 插件。对应 Web 里的浏览器。 Client 是 Host 内嵌的 MCP 协议库，负责跟 Server 用 MCP 语法通信。对应 HTTP 协议栈。 Server 是提供工具或资源的程序——文件系统、数据库、Git、企业内 API。对应 Web 服务器。 (现场演示：Claude Desktop 接一个本地 filesystem-server，让它列目录、读文件、做总结) AI 真的去读了我电脑上的文件——这是从"对话"到"动手"的第一个可感瞬间。

Chatbot vs Agent

前者告诉你"怎么做"，后者直接"做了"

普通 Chatbot

把答案讲给你

  • 你问："帮我订明天去北京的机票"
  • 它答："你可以打开携程，搜索…点击…"
  • 本质：让你去做
  • 只有文字，没有工具
Agent

调工具直接完成

  • 你说："帮我订明天去北京的机票"
  • 它：查航班 → 选最合适 → 调支付 → 返回订单号
  • 本质：分解目标 → 调工具 → 报告结果
  • 由思考决定每一步做什么
关键能力：工具调用 + 多步规划。ReAct 范式 —— Reasoning 一步，Acting 一步，循环往复
Chatbot 和 Agent 看起来差不多，本质完全不同。 你问 Chatbot——"帮我订明天去北京的机票"。它说——"你可以打开携程，搜索 XX、点击 YY"。它把答案讲给你，让你自己去做。 你问 Agent 同一个问题。它直接——查航班 → 选最合适 → 调支付 → 返回订单号。它自己把事做完了。 差别在哪？Agent 有两个 Chatbot 没有的能力——工具调用，加多步规划。 这个范式有个名字叫 ReAct——Reasoning 加 Acting，循环往复。想一步、动一步、看结果、再想再动。 上一章我们讲了 MCP，那是 Agent 的"手"——标准化的工具接口。这一章讲的是 Agent 自己的"脑"——怎么决定下一步做什么。 两个加起来，才是完整的 Agent。

IN DESIGN 15 min
CH.12 · 01 / 03 · 为什么

一个 Agent 不够用

复杂任务要分工，AI 也一样

单 Agent

一个人扛所有事

  • 提示词膨胀 —— 一个 prompt 装不下太多角色
  • 上下文混乱 —— 客服、研发、销售思路撞车
  • 无法并行 —— 一步步串行慢
  • 难以调试 —— 出错不知道哪一环
多 Agent 协作

分工 + 协议沟通

  • 每个 Agent 专精一个角色
  • 各自有独立上下文，思路清晰
  • 可以并行
  • 哪个 Agent 出错容易定位
从一个 Agent 到一组 Agent —— 下一个问题是：他们之间怎么沟通
有了一个 Agent，下一步是多个 Agent。 为什么需要多个？因为单 Agent 扛不住复杂任务——提示词膨胀、上下文混乱、没法并行、出错难调试。 就像公司不会让一个员工同时做客服、研发、销售——分工是必然的。 多 Agent 协作的好处——每个 Agent 专精一个角色，各自有独立上下文，可以并行，哪个 Agent 出错也容易定位。 但下一个问题来了——Agent 多了，他们之间用什么语言沟通？这就是下一页讲的 A2A 协议。
CH.12 · 02 / 03 · A2A 协议

A2A · Agent 之间的开放协议

Google 2025 年发布的开放协议 —— 让不同来源的 Agent 之间能互相通信、协作

  • 定位
    A2A 解决 Agent ↔ AgentMCP 解决的是 Agent ↔ 工具，两个协议是互补关系
  • 机制
    Agent Card · Capability Discovery · Task Lifecycle每个 Agent 公布自己能做什么，其他 Agent 才能找到合作伙伴
  • 支持
    多家公司加入支持2025–2026 期间，IBM、Salesforce、Microsoft、Anthropic 等都有相关支持声明
MCP（Agent ↔ Tool）+ A2A（Agent ↔ Agent）= 完整的 Agent 协作协议栈
A2A 是 Google 2025 年发布的开放协议——让不同来源的 Agent 之间能互相通信、协作。 它和 MCP 是互补关系——MCP 解决 Agent 怎么接工具，A2A 解决 Agent 怎么找伙伴。 A2A 内部有 Agent Card、Capability Discovery、Task Lifecycle 三大机制。简单说就是——每个 Agent 公布自己能做什么，别的 Agent 才能找到合作伙伴。 协议细节我们今天不展开。你们记住一件事——2024 年起，AI 行业有了两套开放协议：MCP 加 A2A。整个 Agent 生态在朝开放、可组合的方向走。
